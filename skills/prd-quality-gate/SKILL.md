# PRD 质量闸门 (Quality Gate)

## Overview

对产品需求文档（PRD）做自动质量初筛，把"文档质量"从主观感受变成可判定、可卡门的结构化评估。输出一份风险报告 + 红黄绿分级，帮助团队在进入开发前过滤掉明显的缺失、模糊和矛盾。

**定位：自动初筛是"减负"不是"替代"** —— 过滤掉明显问题，把人工精力留给真正需要判断的可行性、价值、业务逻辑。

## When to Use

- 用户丢来一份 PRD / 需求文档 / 需求描述，要求"评估质量""看看能不能用""检查一下"时
- 产品评审前需要快速初筛时
- 开发排期前需要确认 PRD 是否达到交付标准时

**不要用于：**
- 代码审查（用 code review skill）
- 技术方案评审（属于架构评审，不是 PRD 质量门禁）
- 纯业务逻辑正确性判断（能查"前后矛不矛盾"，但不能查"业务规则本身对不对"——比如能发现"前文说可取消后文说不可取消"，但不能判断"该不该允许取消"）

## 评估流程

### Step 0: 前置检查 — 语雀 MCP 安装与配置

> 仅当用户通过**语雀链接**提供 PRD 时需要执行本步骤。文件路径或直接粘贴文本的跳过。

**0.1 检测 yuque-mcp 是否已安装**

**优先用通用方式探测**（不依赖具体客户端）：

```bash
# 探测 npx 能否找到 yuque-mcp 包
npx yuque-mcp --help 2>/dev/null
```

- 有输出（显示帮助信息）→ 包已安装，进入 0.3 验证 Token
- 无输出 / 报错 → 未安装，进入 0.2

如果当前运行环境有客户端级 MCP 管理命令，也可辅助确认：

```bash
# 视实际客户端而定，以下为常见示例，非必须
claude mcp list          # Claude Code
# 或检查对应客户端的 MCP 配置文件
```

**0.2 未安装时的安装流程**

1. 引导用户访问 https://github.com/yuque/yuque-mcp-server ，按 README 的 Quick Start 完成安装与配置
2. 安装前需准备：语雀 Token（访问 https://www.yuque.com/settings/tokens 创建）
3. 安装后需重启当前会话（MCP 配置在会话启动时加载）

**0.3 验证 Token 是否有效**

安装完成后，调用语雀 MCP 的只读接口验证连通性：

```
# 调用 yuque MCP 的 search 或 get_user 接口
# 能正常返回 → Token 有效，进入 Step 1
# 返回 401/403 → Token 无效或权限不足，提示用户重新获取
```

**完成标准**：`npx yuque-mcp --help`（或等效的客户端 MCP 列表命令）能确认 yuque-mcp 可用，且调用语雀只读接口能正常返回数据。安装失败时降级为"请用户复制 PRD 文本粘贴进来"。

### Step 1: 获取文档内容

根据用户提供的输入方式获取 PRD 全文：

- **文件路径** → `read_file` 读取（支持 .md / .txt / .docx / .pdf）
- **直接粘贴文本** → 直接使用消息中的文本
- **链接**（语雀文档）→ 先执行 Step 0 确认 yuque-mcp 可用，再用 yuque-mcp 的 `get_doc` 工具获取文档；markdown 正文（body 字段）中的 `![desc](url)` 图片链接需保留用于 C3 原型图比对；抓不到则请用户粘贴文本

**完成标准**：拿到 PRD 的完整文本内容，确认没有截断或遗漏。如文档超过 1.5 万字，分章节处理但需记录跨章节检查的局限。

### Step 2: 按六大维度逐项评估

对下方「评估维度与标准」中的每个检查项逐一判定，结果为：

- ✅ **PASS** — 满足要求
- ⚠️ **WARN** — 部分满足，有风险
- ❌ **FAIL** — 不满足，需补齐

**关键要求**：每个 WARN 和 FAIL 项**必须**记录：
1. **原文引用**：从文档中摘录对应片段（用「」标注）
2. **问题描述**：为什么这是个问题
3. **改进建议**：具体怎么改（给可执行的改写示例，不是"建议补充"）

**完成标准**：六大维度所有检查项均有判定结果；所有 FAIL/WARN 项的原文引用 + 改进建议均已记录，无遗漏。

### Step 3: 判定红黄绿分级

按以下规则定级（Blocker 优先于总分）：

| 级别 | 条件 | 处置 |
|------|------|------|
| 🔴 红 | 有任一 Blocker 项为 FAIL | 打回重写 |
| 🟡 黄 | 无 Blocker FAIL，但总分 < 75 或任一维度得分 < 60 | 带条件放行，限期补齐 WARN 项 |
| 🟢 绿 | 总分 ≥ 80 且无 Blocker FAIL、无维度 < 60 | 放行进入人工评审 |

**Blocker 项（一票否决，不可降级为 WARN）：**
1. A3 — 核心场景/用户故事缺失
2. A4 — 功能需求清单缺失
3. A8 — 验收标准缺失
4. E2 — 范围边界（做什么/不做什么）未定义
5. C4 — 功能间存在逻辑互斥（如 A 要求必填、B 要求同一字段选填）

**完成标准**：给出明确的红/黄/绿判定 + 判定理由（引用触发条件的具体项）。

### Step 4: 输出结构化报告

按「报告格式」输出完整报告。报告是面向产品同学的交付物——要让产品拿到后直接知道改哪里、怎么改。

**完成标准**：报告包含评分明细、Blocker 列表、风险清单（按严重度排序）、Action Items（按优先级排序）。

## 评估维度与标准

### A. 完整性 Completeness —「该有的有没有」

| # | 检查项 | Blocker? | PASS 标准 |
|---|--------|----------|-----------|
| A1 | 背景与目标 | | 说明为什么做、解决谁的什么问题 |
| A2 | 目标用户 | | 明确面向哪类用户（不是"所有用户"） |
| A3 | 核心场景/用户故事 | ⭐ | 主流程清晰，有步骤可追溯 |
| A4 | 功能需求清单 | ⭐ | 按模块拆解，非一整段描述 |
| A5 | 非功能需求 | | 覆盖性能/安全/兼容性中的至少一项 |
| A6 | 异常流与边界条件 | | 失败、空状态、并发、超时等至少提及 |
| A7 | 数据需求 | | 数据来源、存储、隐私合规至少一项 |
| A8 | 验收标准 | ⭐ | 核心功能有可验收的标准 |
| A9 | 优先级标注 | | 有 P0/P1 或等效的优先级划分 |
| A10 | 度量指标 | | 上线后如何衡量成功（至少一个量化指标） |
| A11 | 依赖与前置 | | 依赖其他团队/系统/上线节奏时已说明 |
| A12 | 页面展示要素完整 | | 涉及页面/界面时，字段、label、下拉框选项、报错文案、提示文案、必填项标注等展示要素齐全（见下方检查清单） |

**页面展示要素检查清单**（PRD 涉及页面/界面时逐项扫描，命中缺失即标 WARN/FAIL）：
- **字段**：页面要展示哪些字段（名称、类型、来源）是否逐一列出
- **Label**：每个字段的展示名称是否明确（不是只写"用户名"，而是写清"用户名（手机号）"还是"用户名（邮箱）"）
- **下拉框/选项**：下拉框的待选项是否完整列举，还是只写"常见选项"
- **报错文案**：每个校验失败的报错文案是否写明（不是"提示错误"，而是"手机号格式不正确"）
- **提示文案**：引导性/帮助性提示文案是否写明
- **必填项**：哪些字段必填、哪些选填是否标注
- **空状态**：无数据时页面展示什么
- **按钮/操作**：页面上的按钮及其触发动作是否列出
- **状态切换**：页面有状态变化时（如启用/禁用），各状态的展示差异是否说明
- **展示一致性**：同一字段/label 在文档不同位置（需求描述、验收标准、原型说明）的表述是否一致

### B. 清晰性 Clarity —「能不能无歧义地看懂」

| # | 检查项 | PASS 标准 |
|---|--------|-----------|
| B1 | 无模糊表述 | 无"优化体验""做得好一点"等无具体标准的描述 |
| B2 | 需求可量化 | 验收时能用客观标准判定通过/不通过 |
| B3 | 无主观判断依赖 | 无"界面美观""交互流畅"等纯主观描述 |
| B4 | TBD 密度合理 | 未决项占比低，且每个 TBD 有预计确认时间 |
| B5 | 配图支撑 | 流程/状态/界面有图示辅助（纯文字 PRD 需指出） |
| B6 | 功能点粒度合理 | 同一功能的描述集中在一处，不东一块西一块散落在其他功能描述下 |
| B7 | 上下级归属正确 | 功能层级关系正确——A 功能下的子项不扯到 B 功能的内容；父子归属清晰 |

**功能组织结构检查要点**：
- **不跨功能散落**：A 功能的描述不应一部分出现在 B 功能章节下、另一部分出现在 A 功能章节下。若发现 A 的内容写在 B 下，标 FAIL 并指出应挪到哪。
- **上下级正确**：不能在 A 功能的子项描述中插入属于 B 功能的内容。检查每个子项是否真正从属于其父功能。
- **粒度均匀**：不能 A 功能写了 3 行、B 功能写了 3 页，差异过大且无合理原因。粗粒度与细粒度混搭时需指出。
- **模块边界清晰**：功能模块之间的边界是否明确，有无交叉重叠

**模糊表述扫描清单**（命中即标 WARN/FAIL）：
- 「优化」「提升」「改善」「增强」「完善」+ 无具体指标
- 「合理」「适当」「良好」「流畅」「友好」+ 无客观标准
- 「待定」「TBD」「后续确认」「视情况」+ 无时间或条件

### C. 一致性 Consistency —「自相矛盾 / 与现状冲突」

| # | 检查项 | Blocker? | PASS 标准 |
|---|--------|----------|-----------|
| C1 | 名称前后一致 | | 同一功能/页面/字段/按钮在不同位置的名称完全一致 |
| C2 | 功能描述前后一致 | | 同一功能在不同章节的描述不矛盾（行为、规则、流程一致） |
| C3 | 需求文档与原型图一致 | | 文字描述与原型图/设计稿的内容、字段、交互一致（如有原型图） |
| C4 | 功能间无互斥 | ⭐ | 不同功能之间不存在逻辑互斥或冲突（如 A 要求必填、B 要求同一字段选填） |
| C5 | 术语统一 | | 同一概念全文用同一叫法（含缩写/中英混用） |
| C6 | 展示要素表述一致 | | 同一字段/label/文案在需求描述、验收标准、原型说明中表述一致 |
| C7 | 与既有功能不冲突 | | 不与已上线功能产生矛盾（能判断时） |
| C8 | 与产品目标对齐 | | 需求服务于文档开头声明的业务目标 |

**不一致类型扫描清单**（发现以下任一情况必须明确指出，不可遗漏）：
- **名称不一致**（MEDIUM）：同一功能/页面/字段在文档前文叫"用户中心"，后文叫"个人中心"
- **描述不一致**（HIGH）：前文说"手机号登录"，后文验收标准写"账号+密码登录"
- **文档与原型不一致**（MEDIUM-HIGH）：需求描述写了 5 个字段，原型图上有 7 个；或原型上有按钮但需求没提
- **功能互斥**（Blocker — C4）：功能 A 要求"订单不可取消"，功能 B 要求"用户可随时取消订单"
- **规则冲突**（HIGH）：前文"最多上传 5 张"，后文"最多上传 10 张"
- **状态矛盾**（HIGH）：某处说"默认启用"，另一处说"默认禁用"

**术语检查要点**：不只找完全不同的词，还要注意——
- 缩写混用：「用户ID / UID / 账号ID」
- 中英混用：「支付 / Payment」「订单 / Order」
- 近义词：「模块 / 功能 / 组件」指代不清

### D. 可测试性 Testability —「能不能验收」

| # | 检查项 | PASS 标准 |
|---|--------|-----------|
| D1 | 验收标准存在 | 每条核心需求有对应验收标准 |
| D2 | 验收标准可量化 | 给定输入 → 预期输出，可复现 |
| D3 | 不依赖主观判断 | QA 能独立判定通过/不通过 |
| D4 | 边界条件有验收 | 极端值、空值、并发等场景有预期结果 |

### E. 可行性与边界 Feasibility & Scope —「能不能做、范围清不清」

| # | 检查项 | Blocker? | PASS 标准 |
|---|--------|----------|-----------|
| E1 | 技术可行性 | | 无明显技术障碍（粗判，非详评） |
| E2 | 范围边界明确 | ⭐ | 做什么和不做什么都写了 |
| E3 | 资源/工期合理性 | | 工期与范围大致匹配 |
| E4 | 风险与降级 | | 识别主要风险并有降级方案 |

### F. 价值与对齐 Value & Alignment —「值不值得做」

| # | 检查项 | PASS 标准 |
|---|--------|-----------|
| F1 | 与业务目标对齐 | 需求服务于明确的业务目标 |
| F2 | ROI 粗判 | 投入产出合理（非明显高投入低价值） |
| F3 | 优先级合理性 | P0 项确实是核心价值，非边缘功能 |

## 评分规则

- 每个维度满分 100 分，按 PASS/WARN/FAIL 比例折算：
  - PASS = 该项满分
  - WARN = 该项半分
  - FAIL = 该项 0 分
- 总分 = 六维度等权平均（可根据团队侧重点调整权重）
- **Blocker 项 FAIL 直接触发红色，不看总分**
- 维度内 Blocker 项 FAIL 时，该维度得分上限锁定为 40 分

## 报告格式

输出给用户时使用以下格式（Markdown）：

```markdown
# 📋 PRD 质量评估报告

**文档**：<文档名称/标题>
**评估时间**：<YYYY-MM-DD>
**总体判定**：<🔴红 / 🟡黄 / 🟢绿>
**总分**：<X>/100

## 评分明细

| 维度 | 得分 | 状态 |
|------|------|------|
| A. 完整性 | XX | ✅/⚠️/❌ |
| B. 清晰性 | XX | ✅/⚠️/❌ |
| C. 一致性 | XX | ✅/⚠️/❌ |
| D. 可测试性 | XX | ✅/⚠️/❌ |
| E. 可行性与边界 | XX | ✅/⚠️/❌ |
| F. 价值与对齐 | XX | ✅/⚠️/❌ |

## 🚫 Blocker（一票否决项）

> 列出所有 FAIL 的 Blocker 项。无则显示"无 Blocker"。

- **A3 核心场景** — FAIL
  - 原文：「<引用或标注"全文未找到">」
  - 问题：<说明>
  - 建议：<如何补>

按严重程度（HIGH → MEDIUM → LOW）排序：

   - 问题：无具体指标，无法验收

### MEDIUM
2. **[C2 术语不统一]**
   ...

### LOW
3. ...

## ✅ Action Items

按优先级排序，可直接执行：

- [ ] 补充核心场景/用户故事（Blocker）

- [ ] 统一术语：UID / 账号ID → 用户ID
- [ ] ...

<可选：总体评价、文档亮点、特别需要人工评审关注的点>
```

## Common Pitfalls

1. **只评不引** — 每个 FAIL/WARN 必须引用文档原文。没有引用的判定等于没判，产品不知道改哪里。引用 > 描述 > 感受。

2. **建议太空泛** — "建议补充验收标准"❌ → "建议为'用户登录'补充：输入正确账号密码后 200ms 内跳转首页，错误时显示具体原因"✅。建议必须可执行。

3. **Blocker 放水** — Blocker 项不可降级为 WARN。要么 PASS 要么 FAIL，没有中间态。这是闸门的底线。

4. **长文档漏检** — PRD 超过 1 万字时，按章节逐段扫，避免注意力分散导致后半部分漏检。跨章节的一致性检查（C1/C2）要全文扫完再判定。

5. **术语只找完全不同的词** — 也要注意缩写混用（"用户ID/UID"）、中英混用（"支付/Payment"）、近义词混指（"模块/功能/组件"）。

6. **忘记"不做什么"** — E2 范围边界不只看写了做什么，还要看有没有明确不做什么。缺失"不做什么"= 范围蔓延风险。

7. **把可行性当技术评审做** — E1 只做粗判（有无明显技术障碍），不做详细技术方案评审。那是架构评审的事。

8. **忽略"评估局限"** — 自动初筛能查"有没有写""写得清不清楚"，不能查"写得对不对"。报告末尾应提示需要人工关注的点（如业务逻辑正确性、技术可行性细节）。

9. **页面展示要素漏检** — PRD 涉及页面/界面时，容易只关注功能流程而忽略展示要素（字段、label、选项、文案、必填项）。必须对照「页面展示要素检查清单」逐项扫，不能跳过。

10. **功能组织问题不当回事** — 功能描述散落、上下级归属混乱看似"小问题"，但会导致开发理解偏差和遗漏。A 的内容写在 B 下面 = 开发做 B 时可能漏掉 A 的部分。发现即标 FAIL。

11. **不一致只找术语层面** — 不一致不只是"叫法不同"。最危险的是功能互斥和规则冲突（如一个说可取消一个说不可取消），这类逻辑矛盾必须标 HIGH 并在报告最显眼处提出。

12. **原型图不一致漏检** — 如果 PRD 附带原型图/设计稿链接，必须对比文字描述与原型内容。常见问题：需求写了 5 个字段但原型有 7 个，或原型上有按钮但需求没提。没有原型图时标注"未提供原型图，无法做文档-原型一致性检查"。
